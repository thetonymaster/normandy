## Findings

### F1 — `TokenCounter` is fully broken against the live API — SHOULD-FIX (borderline BLOCKER)

- **Repro:** any `TokenCounter.count_message/2,3`, `count_conversation/2`, or `count_detailed/2` against real Claude returns `{:error, 400 invalid_request_error: "max_tokens: Extra inputs are not permitted"}`. Reproduced across 6 model ids — uniform 400, model-independent.
- **Root cause:** `lib/normandy/context/token_counter.ex:42` and `:66` add `"max_tokens" => 1` to the count-tokens payload. The Anthropic `/v1/messages/count_tokens` endpoint forbids `max_tokens`.
- **Blast radius:** isolated — `TokenCounter` is not called anywhere in `lib/` (only its own moduledoc examples). Compaction (#4) and `WindowManager` use the local `estimate_tokens/1` heuristic + `ModelCatalog.context_window/1`, so the turn loop / compaction are NOT affected. But `TokenCounter` is a public, documented API (listed in CLAUDE.md), and it is the reason all 13 token-counter tests are skipped.
- **Suggested fix:** drop `"max_tokens"` from both count-tokens payloads.
- **Status:** ✅ FIXED — `"max_tokens"` removed from both payloads (`token_counter.ex`). Verified live: `count_message` now returns `{:ok, %{"input_tokens" => 11}}`. The 13 token-counter tests are enabled (now `:integration`) and pass live.

### F1b — `TokenCounter` default model was retired — FIXED (bundled with F1)

- **Issue:** `count_message/2` and `count_conversation/2` defaulted to `claude-3-5-sonnet-20241022`; bare calls (no explicit model) would 404 against the current API.
- **Status:** ✅ FIXED — default changed to `claude-haiku-4-5-20251001`. No internal callers, so blast radius is external consumers relying on the default (strict improvement: retired → current).

### F2 — `NormandyIntegrationHelper` default model is retired — SHOULD-FIX (test infra)

- **Repro:** `NormandyIntegrationHelper.create_real_agent/1` with no `:model` → messages endpoint `404 not_found: "model: claude-sonnet-4-20250514"`. Every `test/normandy_integration/*` test that relies on the default 404s; the `normandy_integration` CI job (if run) is red.
- **Root cause:** `test/support/normandy_integration_helper.ex:53` hardcodes the retired `claude-sonnet-4-20250514`, with no env override. (`IntegrationHelper` defaulted to the current `claude-haiku-4-5-20251001` — the two helpers had diverged.)
- **Suggested fix / agreed enabler:** added `default_model/1` (honors `NORMANDY_TEST_MODEL`, current fallback) to both helpers; routed the one stale per-test pin through it. See "Working-tree changes".
- **Status:** addressed in working tree (test infra) to enable the campaign; underlying shipped issue reported here.

## Observations

### O1 — 13 `TokenCounter` tests are `@tag :skip` (`test/context/token_counter_test.exs:72–251`)

Root cause is a combination: (a) F1 makes the live endpoint call fail; (b) no injection seam — the `MockClaudio` module defined in the test (lines 10–42) is dead code, never wired into `TokenCounter`, which calls `Claudio.Client.new/1` + `Claudio.Messages.count_tokens/2` directly; (c) the fixtures pin retired models (`claude-3-5-sonnet-20241022`, `claude-3-opus-20240229`). Enabling them requires fixing F1 first, then either a mock seam or a real key + current model.

**Status:** ✅ RESOLVED — F1/F1b fixed; the 13 are now `:integration` tests using a real client + current model and pass live (15/15 in the file, incl. 2 local WindowManager tests). The two edge-case tests (empty message / empty memory) now assert the real endpoint contract (`{:error, 400}`) instead of the mock's fictional `{:ok, >= 0}`.

### O2 — dead `MockClaudio` module in `token_counter_test.exs` — NIT

Defined (lines 10–42) but never referenced. Either wire it in (gives `TokenCounter` a testable seam, removing the API dependency) or delete it.

**Status:** ✅ RESOLVED — deleted (the tests now exercise the real endpoint).

### O3 — test-file compile warnings in `test/normandy_integration/` — NIT

Surfaced when the `normandy_integration` suite compiled (the `--warnings-as-errors` baseline only compiles `lib/`, so these are invisible to it):

- `test/normandy_integration/multi_agent_test.exs:4` — unused alias `BaseAgent`.
- `test/normandy_integration/dsl_comprehensive_test.exs:140,145,186` — unused variable `current_step_input`.
- `test/normandy_integration/dsl_comprehensive_test.exs:131,152,165,177` — **duplicate `defmodule`**: `ResearchWorkflow`, `ParallelAnalysisWorkflow`, `RaceWorkflow`, `ConditionalWorkflow` are each defined twice; the second definition shadows the first ("clause cannot match because a previous clause matches"). The tests pass, but the shadowed copies are dead and the intent is ambiguous.
- Numerous unused-variable / unused-alias warnings across `test/integration/*` (`agent`, `response2`, `generator`, and aliases `Processor`, `AgentProcess`, `AgentSupervisor`, `SequentialOrchestrator`, `ParallelOrchestrator`, `StringManipulator`, `BaseAgent`). Cosmetic; same cleanup bucket.

### O4 — `AgentProcess :server` returns bare content, inline returns a struct/map — OBSERVATION (confirm intent)

The live Phase 6 smoke: `AgentProcess.run(pid, "…")` in `:server` mode returned `{:ok, "pong"}` (the raw content string), whereas the inline path surfaces a response carrying `:chat_message`. The user message still persisted to the store as `%{chat_message: "…"}` and `get_agent` reconstructed it correctly. This may be intentional (`convert_turn_output/3` had a Phase 6 fix for non-`chat_message` schemas), but the inline-vs-`:server` result-shape asymmetry is worth a deliberate confirm before consumers depend on either shape. Not reproduced as a failure.
