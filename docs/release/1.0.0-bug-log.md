# 1.0.0 E2E Bug-Hunt — Bug Log & Go/No-Go

## Severity rubric
- **Blocker** — data loss; corruption on resume/rehydrate; guardrail fail-open
  (disallowed input passes); crash on a documented path; credentials persisted.
  **Any open Blocker = no-go.**
- **Major** — wrong behavior under an edge case, recoverable and not
  security-relevant. Triaged case-by-case; may ship as a documented known-issue.
- **Minor** — cosmetic, logging, message text.
- **Flaky** — timing-dependent; logged for investigation, not an automatic blocker.

### BUG-001: MnesiaDistributedTest hangs on node join — mnesia_controller.connect_nodes blocks until 60 s timeout
- **Severity:** Blocker
- **Repro:** `mix test --include distributed --include postgres --include redis 2>&1 | tee evidence/04-distributed.txt`
- **Observed:** Test "a session written on one node is replicated and readable on another" (Normandy.Behaviours.SessionStore.MnesiaDistributedTest) timed out after 60 000 ms. Last frame before timeout: `(mnesia 4.25.2) mnesia_controller.erl:474: :mnesia_controller.connect_nodes/2` triggered by `{:ok, _} = :mnesia.change_config(:extra_db_nodes, [peer_node])` on line 32 of the test. (evidence: evidence/04-distributed.txt:2957)
- **Expected:** Mnesia connects to the in-VM peer node and the session written on node A is readable on node B within the timeout.
- **Root-cause theory:** THEORY — When the `:distributed` suite boots a `:peer` node in-VM, Mnesia on the peer is either not started or not schema-compatible with the primary node's Mnesia instance, causing `mnesia_controller:connect_nodes/2` to block indefinitely waiting for schema negotiation that never completes. The peer start-up path may not be calling `:mnesia.start()` or setting the correct Mnesia dir before the test calls `:mnesia.change_config/2`.

### BUG-002: All 10 MnesiaTest cases hang at setup — mnesia_schema:schema_transaction deadlocks when run alongside distributed suite
- **Severity:** Blocker
- **Repro:** `mix test --include distributed --include postgres --include redis 2>&1 | tee evidence/04-distributed.txt`
- **Observed:** Every test in Normandy.Behaviours.SessionStore.MnesiaTest timed out after 60 000 ms in setup (not in the test body). Stacktrace shows the timeout inside `mnesia_schema.erl:684: :mnesia_schema.schema_transaction/1` ← `Mnesia.create_table/4` ← `Mnesia.create_tables/1` ← `Mnesia.new/1` ← `mnesia_test.exs:5 __ex_unit_setup_0/1`. All 10 affected tests are in the same module. (evidence: evidence/04-distributed.txt:3304)
- **Expected:** Each MnesiaTest creates its own Mnesia instance in setup and completes all 10 tests within the timeout.
- **Root-cause theory:** THEORY — BUG-001 leaves Mnesia's global schema lock held (or a partial schema transaction in flight) on the primary BEAM node after the 60 s test process is killed. Subsequent test processes that call `:mnesia.create_table/4` attempt to acquire the same schema lock and block indefinitely. Because all 10 MnesiaTest setup calls serialise through that global lock, all 10 tests cascade into the same timeout. This is a cross-test Mnesia global-state leak: the distributed test must clean up Mnesia (stop it, clear its dir) before MnesiaTest setup runs.
