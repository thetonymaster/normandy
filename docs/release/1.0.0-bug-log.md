# 1.0.0 E2E Bug-Hunt — Bug Log & Go/No-Go

## Severity rubric
- **Blocker** — data loss; corruption on resume/rehydrate; guardrail fail-open
  (disallowed input passes); crash on a documented path; credentials persisted.
  **Any open Blocker = no-go.**
- **Major** — wrong behavior under an edge case, recoverable and not
  security-relevant. Triaged case-by-case; may ship as a documented known-issue.
- **Minor** — cosmetic, logging, message text.
- **Flaky** — timing-dependent; logged for investigation, not an automatic blocker.

### BUG-001: MnesiaDistributedTest contaminates / is contaminated by shared-BEAM Mnesia state → 60s hang at change_config
- **Severity:** Major (TEST-INFRA — **not** a product blocker)
- **Repro:** `mix test --include distributed --include postgres --include redis 2>&1 | tee evidence/04-distributed.txt`
- **Observed:** Test "a session written on one node is replicated and readable on another" (Normandy.Behaviours.SessionStore.MnesiaDistributedTest) timed out after 60 000 ms at `(mnesia) mnesia_controller.erl:474: :mnesia_controller.connect_nodes/2`, triggered by `:mnesia.change_config(:extra_db_nodes, [peer_node])` (test line 32). (evidence: evidence/04-distributed.txt:2957)
- **Expected:** Mnesia clusters with the peer and the session written on node A is readable on node B.
- **Root cause:** **VERIFIED (controller experiment).** Run in isolation in a properly-distributed node (`elixir --name primary@127.0.0.1 -S mix test test/behaviours/session_store/mnesia_distributed_test.exs --include distributed`), the test **PASSES in 0.1 s** — cross-node Mnesia clustering + replication works; the shipped #37 driver is sound (H3 driver-bug REFUTED). The combo-run hang is contamination: other same-BEAM Mnesia tests leave the primary node with tables in the default schema, so `change_config(:extra_db_nodes)` deadlocks the schema merge with the fresh peer. Test-only; no product runtime path hits this.
- **Fix:** resolved in this commit (75f206d)

### BUG-002: All 10 MnesiaTest cases hang at create_table whenever the run includes `--include distributed`
- **Severity:** Major (TEST-ISOLATION) — plus an OPEN product question (see below)
- **Repro:** `mix test --include distributed --include postgres --include redis 2>&1 | tee evidence/04b-distributed-after-fix.txt`
- **Observed:** Every test in Normandy.Behaviours.SessionStore.MnesiaTest times out after 60 000 ms in setup at `mnesia_schema.erl:684: :mnesia_schema.schema_transaction/1` ← `Mnesia.create_table/4` ← `mnesia_test.exs:5`. (evidence: evidence/04b-distributed-after-fix.txt:6547)
- **Expected:** Each MnesiaTest creates its Mnesia tables in setup and completes.
- **Root cause:** **PARTLY FALSIFIED — NOT a cascade of BUG-001.** After BUG-001's distributed-Mnesia test was fixed (it now passes in the full combo), MnesiaTest STILL hangs (10/10) — see evidence/04b. So the earlier "leaked schema lock from the hung distributed test" theory is wrong. Confirmed facts: MnesiaTest passes with **zero** Mnesia timeouts in floor/postgres/redis (non-distributed BEAM), and hangs only when `--include distributed` is present. `create_table` hangs in a Mnesia schema transaction. Leading theory (UNVERIFIED): once the BEAM is distributed and Mnesia has been clustered to a now-dead peer (`extra_db_nodes`), `create_table`'s schema transaction blocks coordinating with the dead db_node. **OPEN PRODUCT QUESTION:** if `Mnesia.create_tables/1` (and possibly `append`'s `:mnesia.transaction`) hang on a clustered node after a peer dies, that could be a real distributed-Mnesia-store robustness issue, not merely test isolation. Needs a focused experiment to separate test-only from product.
- **Product impact: NONE (VERIFIED).** Direct experiments (/tmp/mnesia_peer_death.exs) prove the shipped driver is robust: clustering ✓, replication ✓, `append` after peer death ✓, `create_tables` after peer death ✓, `:mnesia.stop()`+`start()` cleanly drops the dead peer ✓. No product runtime path hangs. This is a **test-suite isolation** defect only, confined to the full `--include distributed` combo (CI never runs these gated suites together).
- **Root cause: VERIFIED (the disc_copies lead above was WRONG).** The combo BEAM starts non-distributed as `:nonode@nohost`; early floor/Mnesia tests create a Mnesia schema recording `:nonode@nohost` as a db_node. The first `:distributed` test then calls `:net_kernel.start/1`, **renaming the node** to `primary@127.0.0.1` and orphaning the old schema's `:nonode@nohost` db_node. Every subsequent `MnesiaTest` `create_table` runs a `schema_transaction` that blocks trying to coordinate with the phantom `:nonode@nohost`. Minimal repro: `mix test test/behaviours/session_store/mnesia_test.exs test/agents/turn/horde_distributed_test.exs --include distributed --include postgres --include redis --timeout 8000 --max-failures 1`. Not Mnesia-clustering or disc_copies related; the contaminator is the node rename, from any distributed test.
- **Fix: RESOLVED (commit 1668ade).** Test-only: a `setup` in `mnesia_test.exs` (before `use SessionStoreContract`) resets Mnesia (`:mnesia.stop()` → `:mnesia.delete_schema([node()])` → `:mnesia.start()`) so each test's schema reflects the current node name. No `lib/` change. Verified: full combo `1377 tests, 0 failures` (subagent); controller re-confirming in evidence/04c-distributed-final.txt.
- **Status:** FIXED (test-only). Product driver verified sound throughout.

### BUG-003: MnesiaDistributedTest setup crashes (MatchError) when node is not already distributed
- **Severity:** Minor (TEST-INFRA)
- **Repro:** `mix test test/behaviours/session_store/mnesia_distributed_test.exs --include distributed` (node not pre-named)
- **Observed:** `** (MatchError) no match of right hand side value: {:ok, #PID<...>}` at test line 11. The setup matches `:ok = :net_kernel.start([:"primary@127.0.0.1", :longnames])`, but `:net_kernel.start/1` returns `{:ok, pid}`. (controller-reproduced, isolation run, 0.02 s fail)
- **Expected:** Setup starts distribution and proceeds. (Masked in the combo run because an earlier distributed test already started distribution, so `unless Node.alive?()` skipped line 11.)
- **Root cause:** **VERIFIED.** Wrong match on `:net_kernel.start/1` return shape. Test-only.

### BUG-004: RedisDistributedTest setup has the same net_kernel.start match bug — FLAKY (seed-dependent)
- **Severity:** Major (TEST-INFRA, flaky) — intermittent suite/CI breakage
- **Repro:** `mix test test/behaviours/session_registry/redis_distributed_test.exs --include distributed` (deterministic when this test starts distribution); or the full combo on a seed where this test runs first (e.g. observed in evidence/04c-distributed-final.txt:2478).
- **Observed:** `** (MatchError) no match of right hand side value: {:ok, #PID<...>}` at `redis_distributed_test.exs:14` — `unless Node.alive?(), do: :ok = :net_kernel.start([...])`. Fires only when this is the FIRST `:distributed` test to start distribution (so `:net_kernel.start` actually runs and returns `{:ok, pid}`); on other seeds an earlier test already started distribution and the `unless` skips it. Found by an independent re-run (the fix-subagent's seed didn't trigger it).
- **Root cause:** **VERIFIED.** Same class as BUG-003 — `:ok = :net_kernel.start/1` instead of `{:ok, _}`. The five sibling distributed tests already use `{:ok, _}`; this file was the lone holdout.
- **Fix: RESOLVED.** One-line: `:ok =` → `{:ok, _} =` (matches the sibling pattern). Verified: standalone `2 tests, 0 failures`; full combo `1377 tests, 0 failures` across seeds 0/1/7/42.
- **Fix:** resolved in this commit (75f206d)
